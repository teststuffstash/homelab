## My stated principles (verbatim, from the NTD 2024 "Best Current Practices" slide)

> - Git MUST be the single source of truth
>   - Merge conflicts are resolvable
>   - Immutable environments are stored in git
>   - No "blobs" in pull requests and code reviews
>   - State is repeatable from a tag
> - Everybody MUST have their own test environment
> - Test environments MUST work offline
> - Out of the Box Experience MUST be prioritised over CI considerations
> - Everything MUST be open source and replaceable
> - Test data MUST NOT be hidden
> - Requirements MUST have unique identifiers
> - There SHOULD be a maximum of 5 E2E tests

> _Note on "blobs":_ this means **changes that produce huge, unreviewable diffs** — e.g. a
> one-line SoapUI edit that regenerates 30 000 lines of XML — **not** binary files. The fix
> is **determinism**: everything in the repo must serialize stably and in canonical order
> (a YAML list of IPs sorted smallest→largest, stable key ordering, no autogenerated churn)
> so a small logical change makes a small diff.

These are the canonical source; the lens below is how they apply to infra decisions.

## Principles / the decision lens

When weighing options, prefer the choice that scores better on these — and call it out
when a decision trades one of them away:

1. **Git is the single source of truth** — taken literally. Every system MUST be:
   - **(a) Modifiable only from git** — *no clicking buttons, ever.* Experimenting is the
     sole exception, and it only counts once it's redone properly as Git/IaC.
   - **(b) Reproducible on different hardware** — any system recoverable from git. If
     lightning fries 3 servers, I can `apply` onto 3 rented **AWS EC2** instances in minutes
     while new hardware ships. ⇒ keep configs hardware/cloud-agnostic.
   - **(c) Sharable** — anyone (incl. future me) can clone the repo, buy hardware, and stand
     up the identical setup.

   **Data is the only exception:** it lives in object storage (**S3**), and the **bucket ID
   is in git** — so the system can *"boot from git"*: everything needed to recover, *including
   where the data is*, is version-controlled. (Drives: OpenTofu, Talos machine-config,
   Matchbox-in-git, SOPS for secrets.)
2. **Deterministic, reviewable diffs** — a small logical change MUST yield a small diff.
   Canonical ordering (IP lists sorted ascending), stable serialization, no autogenerated
   churn ("blobs"). Applies to every generated/list config: Matchbox MAC table, inventories,
   Talos/Helm values.
3. **Code/config over GUI/click-ops** — state lives in version control, not in a web UI.
4. **Local-first / offline-capable** — must work on my hardware without a SaaS dependency.
   (Why we chose DIY netboot over Omni, kind for local demos.)
5. **Open-source & replaceable** — no lock-in, nothing I can't self-host or swap out
   (OpenTofu not Terraform, DIY not Omni SaaS). Favor modern standards: Gateway API over
   legacy Ingress, eBPF (Cilium), buildpacks over hand-rolled Dockerfiles.
6. **Out-of-the-box experience first** — prioritise a clean `git clone → it runs` over CI
   convenience; a colleague should be able to reproduce it.
7. **Teachable / demoable** — could this become a clean talk or a thing a colleague can
   reproduce? Favor clarity over cleverness.
8. **Budget-conscious** — secondhand / AliExpress hardware, no enterprise BMC. Solve
   remote-management with vPro/AMT, WoL, smart plugs — not by buying IPMI servers.
9. **Public by default** — this is a **public repo**, so: no secrets in git, write it like
   someone will read it.

## Standing constraints for the homelab

- **No IPMI** on the cheap x86 boxes → remote control via vPro/AMT + WoL + Home Assistant
  smart plugs (see `ROADMAP.md`).
- Mixed, non-uniform hardware. The Proxmox host is an AliExpress **X99 / Xeon E5-2680 v4**;
  it has **no iGPU** but now has a **weak discrete GPU fitted so it POSTs/boots** — fine as a
  fixed hypervisor, still not a GPU-less plug-in. The general fleet rule stands: GPU-less
  plug-in nodes need iGPU or BMC video.
- Single-operator lab: optimize for low day-2 toil and easy rebuild over high availability.
